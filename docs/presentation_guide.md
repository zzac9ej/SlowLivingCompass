## 投影片 3：市場痛點分析

**標題：** 現有的解決方案都不夠好

**講稿：**
> 「現在市面上有哪些解法？Google Maps——200 間餐廳，你還是不知道選哪間。轉輪盤 App——太隨機，完全不考慮大家的偏好。投票工具（如 Slido）——太正式，而且還是要有人先提出選項。我們的產品不一樣：**系統自動出題、每個人匿名投票、命運幫你做決定。**」

**建議放（表格）：**

| 解法 | 問題 |
|------|------|
| Google Maps | 資訊過載，選擇更困難 |
| 轉輪盤 App | 純隨機，不考慮偏好 |
| LINE 投票 | 需要有人先想選項 |
| **佛系揪團** | ✅ 系統出題 + 匿名盲測 + 自動開獎 |

**標題：** 把手機遞給下一個人 — Pass & Play 機制

**講稿：**

**建議放：**

## 投影片 7：功能展示 — Step 3 開獎動畫

**標題：** 最刺激的一刻 — 3-2-1 揭曉！

**講稿：**
> 「這是整個產品最有記憶點的環節。投票結束後，畫面不是直接跳出結果，而是先出現一個大大的倒數：3...2...1...🎉，伴隨著彩色紙花從天而降，最終勝出的品項以彈跳動畫進場。這個儀式感讓『吃什麼』這件小事變成了一個**小型派對**。之後系統會自動導向 Google Maps，搜尋你附近所有該品項的店家。」

**建議放：**

- 結果揭曉截圖：🎉 + 紙花 + 勝出品項（例如「臭豆腐」）
- Google Maps 搜尋結果截圖

## 投影片 8：技術架構

**標題：** 怎麼做到的？

**講稿：**
> 「整個產品基於 Blazor WebAssembly 開發，所有邏輯都在瀏覽器端執行，不需要後端伺服器。投票計票使用 Dictionary 即時統計，平手時由 Random 亂數仲裁——貫徹『不讓人做決定』的理念。部署在 GitHub Pages 上，零伺服器成本。」

**建議放：**

| 層級 | 技術 | 說明 |
|------|------|------|
| 前端框架 | Blazor WebAssembly | C# 編譯為 WASM，瀏覽器執行 |
| UI 元件 | MudBlazor | Material Design 元件庫 |
| 定位 | Geolocation API | 瀏覽器原生 GPS |
| 資料 | places.json + Dictionary | 靜態資料 + 記憶體計票 |
| 動畫 | CSS Keyframes | confetti、bounce-in、fade |
| 導航 | Google Maps URL Scheme | 免 API Key，直接跳轉 |
| 部署 | GitHub Pages | 零成本靜態託管 |

- **AI 驅動開發**：本產品全程使用 AI 輔助開發，大幅提升從 PRD 到成品的效率
