## 一、執行摘要與產品願景

### 背景與問題

每個人都遇過這樣的對話：

```
A：「今晚吃什麼？」
B：「都可以」
C：「隨便」
D：「你決定」
A：「......」
```

這不是溝通問題，而是**決策機制的問題**。

提出建議的人承擔了三個隱形風險：
1. 選得不好被嫌
2. 有人根本不想吃那個
3. 之後每次都換你決定

最安全的策略：**不表態**。

現有工具（Google Maps、LINE 投票、轉輪盤 App）的共同盲點是——要麼提供更多資訊加重選擇困難，要麼仍然需要「有人先提出選項、有人出來承擔責任」。

### 產品定位

**佛系揪團**是一款讓群體在 60 秒內完成飲食決策的決策輔助工具。

> **你負責聚，我負責選。**
> 核心理念：No Pressure Decision

## 二、商業目標與成功指標

### MVP 核心目標
驗證「盲測投票 + 開獎儀式」機制的用戶接受度，驗證使用者願意接受「系統決定結果」的決策方式。

### 成功指標

| 指標 | 說明 | MVP 目標值 |
|------|------|------------|
| **North Star Metric** | 單機輪流完整投票流程完成次數（每週） | ≥ 10 次 |
| 流程完成率 | 啟動後完成開獎的比例 | ≥ 70% |
| 分享觸發行為 | 開獎後拍照/截圖的估算比率 | 每 10 次 ≥ 3 次 |
| Day 1 留存率 | 次日回訪比率 | ≥ 30% |

## 三、目標受眾與使用場景

### 核心 Persona
**Alex，24 歲，東華大學研究生**
- 每週都和同學聚餐 2–3 次
- 每次「去哪？吃什麼？」都要花 20 分鐘
- 沒人想提議，怕承擔選錯的責任
- 希望有工具能幫大家「公平但快速」地做決定

### 使用場景

| 場景 | 人數 | 情境 |
|------|------|------|
| 宿舍朋友臨時聚餐 | 3–4 人 | 大家都在同一個地方，傳手機輪流投票 |
| 下課後想找點心 | 2–3 人 | 選「甜點」類別，60 秒決定喝什麼 |
| 家庭聚會決定晚餐 | 4–6 人 | 長輩小孩都能參與，操作零門檻 |

## 四、核心功能與使用者故事

### Epic
讓群體在最短時間內完成飲食決策，且沒有任何人需要承擔責任感。

---

### AC1：類別選擇
- **Given** 使用者啟動「佛系揪團」功能
- **When** 使用者選擇聚餐類型
- **Then** 系統提供三大類選項：🍖 正餐 / 🍢 小吃宵夜 / 🍰 甜點下午茶
- **And** 每類預設 14 個品項，系統從中**隨機抽取 8 項**，確保每次體驗不重複

---

### AC2：Pass & Play 盲測投票

- **Given** 類別已選好，投票卡片已產生（共 8 張）
- **When** 每位朋友輪流拿過手機，對每個品項選擇 ❤️（喜歡）或 ✖️（跳過）
- **Then** 每人**僅看到自己的畫面**，看不到其他人的投票結果
- **And** 進度條顯示「探索進度 X/8」，引導使用者流暢完成
- **And** 每換下一位朋友時，畫面顯示「請遞給第 N 位知己」提示

**設計亮點：消除從眾效應**
傳統公開投票會讓後面的人受前人影響。盲測機制確保每個人誠實表達真實偏好，讓最終結果更準確反映群體共識。

---

### AC3：開獎動畫（核心體驗）

- **Given** 所有人皆完成投票
- **When** 最後一人確認送出
- **Then** 畫面顯示大倒數：**3 → 2 → 1 → 🎉**
- **And** 彩色紙花從天而降，獲得最多 ❤️ 的品項以彈跳動畫進場
- **And** 顯示勝出品項名稱與描述
- **And** 提供一鍵跳轉 Google Maps 搜尋附近相關店家

**設計哲學：把決策變成派對**
結果不是冷冰冰地跳出答案，而是透過儀式感讓這個瞬間成為朋友間的共同記憶。

---

### AC3-1：平手仲裁（Edge Case）

- **Given** 有兩個以上品項獲得相同票數
- **When** 系統計票完成
- **Then** 系統以 C# `Random` 從平手品項中亂數選出一個
- **And** 仍照正常流程顯示開獎動畫，強化「命運做了決定」的體驗
- **And** 徹底貫徹「拒絕讓使用者做決定」的設計哲學

## 五、系統架構

### 技術棧

| 層面 | 技術 | 說明 |
|------|------|------|
| 框架 | Blazor WebAssembly (.NET 10) | C# 編譯執行於瀏覽器，不需後端 |
| UI | MudBlazor v9 | Material Design 元件庫 |
| 計票邏輯 | C# `Dictionary<string, int>` | 本機記憶體即時統計每個品項得票 |
| 亂數仲裁 | C# `Random` | 平手時公正選出結果 |
| 動畫 | Pure CSS Keyframes | confetti、bounce-in、倒數特效 |
| 地圖 | Google Maps URL Scheme | 無需 API Key，直接跳轉搜尋 |
| 部署 | GitHub Pages + GitHub Actions | 自動 CI/CD，零伺服器成本 |

### 資料結構

投票品項以靜態方式預載，共三類：

| 類別 | 品項數 | 範例 |
|------|--------|------|
| 正餐 | 14 | 牛肉麵、火鍋、拉麵、壽司、熟炒、滷肉飯… |
| 小吃宵夜 | 14 | 臭豆腐、鹹酥雞、雞排、滷味、蚵仔煎… |
| 甜點下午茶 | 12 | 咖啡、珍奶、蛋糕、豆花、鬆餅、冰品… |

每次投票從對應類別隨機抽 8 項，確保重複使用的新鮮感。

### 非功能性需求

| 需求 | 規格 |
|------|------|
| 無後端 | 所有邏輯在瀏覽器端執行，不需伺服器 |
| 零安裝 | 純 Web App，掃 QR Code 或開連結即可使用 |
| 啟動速度 | 首次 WASM 載入 < 5 秒 |
| 操作回應 | 動畫、投票互動 < 100ms |
| 隱私 | 投票資料不離開裝置，不蒐集個人資訊 |
